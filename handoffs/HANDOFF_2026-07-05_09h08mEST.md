## Decisions / host-specific facts locked this session
- **Dangling gotcha diagnostic differs on this host.** Apple clang (aliased `g++`) accepts `-Werror=return-local-addr` and emits `reference to stack memory associated with local variable 'local' returned [-Werror,-Wreturn-stack-address]`. The subject test therefore asserts `"reference to stack memory" in html OR "return-local-addr" in html OR "reference to local" in html` so it stays green on both clang and real GNU g++. The compile correctly fails with `error_kind == "compile"`.
- **`sf_memmap` parse fix (deviation from plan):** the plan's inline `(const void*)&main < (const void*)&g_seed` inside `printf(...)` tripped clang's `<`-as-template-bracket parse (`expected '>'`). Fixed by loading all five addresses into `uintptr_t` locals (via `reinterpret_cast`, `#include <cstdint>`) before comparing. Output strings unchanged; all four ordering booleans print `1` on this host.
- **`extra_compile_flags` wiring:** `topic_yaml._topic` forwards keys via an explicit per-key allow-list (not `**kwargs`) — the new field was added there. `_compile_one` now yields `[]` (not `None`) for the sanitize branch, appends `topic.extra_compile_flags`, and passes `extra_flags or None` to `compile_and_run(source, ..., extra_flags=)`.
